Upgrade the MyOrder.fun receipt and print system into a production-grade branded print engine.

Goal:
Replace the basic plain-text receipt logic with a structured, reusable print rendering system that supports:
1. branded customer receipts
2. starting inventory receipts
3. ending inventory / clock-out sheets
4. future label printing
5. preview + test rendering before printing

The upgraded system must work with the existing print bridge and current printer routing.

Core requirements:
- Keep thermal-printer compatibility
- Keep outputs text-first by default
- Support logo/header branding
- Support multiple receipt template types
- Support printer width awareness
- Keep templates editable and modular
- Do not break the current bridge flow

Build this in a clean modular structure.

1. Create a dedicated print module
Add a print module such as:

server/print/
  index.ts
  logo.ts
  formatter.ts
  widths.ts
  renderer.ts
  preview.ts
  templates/
    customerReceipt.ts
    inventoryStart.ts
    inventoryEnd.ts
    label.ts
  assets/
    logo.txt

Responsibilities:
- logo.ts: load and fit printable logo
- widths.ts: paper width rules
- formatter.ts: helper functions for alignment and spacing
- renderer.ts: converts structured receipt data into printable text
- preview.ts: preview/debug helpers
- templates/: specific templates per print type

2. Add a structured print renderer
Do not build receipts by concatenating ad hoc strings everywhere.
Instead create a shared rendering layer that supports elements like:
- centered text
- divider lines
- blank spacing
- left/right key-value lines
- section headings
- item rows
- totals
- footer notes

Suggested internal shape:
- type: "text" | "center" | "divider" | "kv" | "itemRow" | "spacer"
- text
- left
- right
- style
- emphasis

Then have a renderer convert those blocks into final text based on printer width.

3. Add printable width support
Support thermal widths:
- 58mm
- 80mm

Suggested widths:
- 32 chars for 58mm
- 42 or 48 chars for 80mm

Renderer should:
- read printer width from printer config if available
- default safely if not
- keep alignment correct for both sizes

4. Add formatter helpers
Create reusable helpers such as:
- centerText(text, width)
- divider(width, char = "=")
- kvLine(left, right, width)
- wrapText(text, width)
- itemLine(name, qty, unitPrice, total, width)
- fitPrintableLogo(logo, width)

Make sure long item names wrap correctly without breaking totals alignment.

5. Add brand logo support
Use the provided Alavont ASCII logo as the print header.
Do not hardcode it inline in templates.
Load it from:
- server/print/assets/logo.txt

Behavior:
- include logo at top of all branded receipts
- fallback to "ALAVONT THERAPEUTICS" if file missing
- fit logo to current printer width
- allow easy replacement later

6. Upgrade customer receipt template
Customer receipts should include:
- logo
- receipt title
- order id
- date/time
- operator if applicable
- payment status/method if appropriate
- line items
- subtotal
- tax if present
- total
- discreet footer text
- optional mapping of Alavont / Lucifer Cruz names if configured

Structure example:
[logo]
================================
SECURE ORDER RECEIPT
================================
Order ID: 123
Date: ...
Operator: ...
--------------------------------
Item lines
--------------------------------
Subtotal
Tax
TOTAL
================================
Thank you / discreet footer

7. Upgrade starting inventory receipt template
Starting inventory printout should include:
- logo
- title
- operator
- shift/date
- grouped inventory sections
- row items with starting qty and unit
- signature/verification line if needed

Must render from the structured Business Sitter inventory template, not the old freeform row model.

8. Upgrade ending inventory / clock-out template
Ending inventory / clock-out printout should include:
- logo
- title
- operator
- date/time
- grouped sections
- per-item:
  - start qty
  - sold qty
  - end qty
  - unit
- totals / petty cash if configured
- discrepancy / notes area if useful

9. Add label template support
Even if labels are basic for now, create a separate label template path so receipt logic and label logic are not mixed.

Suggested:
- templates/label.ts

10. Add print preview support
Create admin/dev preview routes or utilities so receipt text can be previewed before sending to the printer.

Examples:
- GET /api/print/preview/customer/:orderId
- GET /api/print/preview/inventory-start/:shiftId
- GET /api/print/preview/inventory-end/:shiftId

Preview response can be:
- plain text
- JSON with rendered blocks and final text

11. Improve test print flow
The test print system should not depend on a fake order_id that breaks DB inserts.
Fix test-print backend flow so:
- test print jobs do not require a real order
- order_id may be null for test jobs
- or use a separate diagnostics path for test print logs

Return clean UI messages, not raw SQL/stack traces.

12. Add dynamic printer selection
Do not rely on one hardcoded printer queue in the bridge.
Upgrade integration so the print request can specify:
- printer name
- copies
- job type

Receipt and label printers must be selectable independently.

13. Add print job logging
Track:
- job type
- template type
- selected printer
- bridge URL
- status
- error message
- created at / printed at

Keep admin-safe error messages in UI.
Log detailed technical errors server-side only.

14. Add settings for receipt appearance
Add admin-configurable print settings:
- include logo toggle
- include Lucifer Cruz mapping toggle
- paper width default
- footer message
- show operator name toggle
- show discreet notice toggle

15. Keep templates editable
Make footer text and some title strings configurable without deep code edits if practical.

16. Preserve existing bridge behavior
Do not break:

- receipt printer config
- label printer config
- business sitter inventory print flow
